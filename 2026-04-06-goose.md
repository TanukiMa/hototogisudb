## Problems identified in the **mozc4med_dict** package (and sub‑directories)

| File / Area | Issue | Why it matters (according to the CLAUDE.md spec) |
|-------------|-------|------------------------------------------------|
| **`mozc4med_dict/importers/base.py`** | 1️⃣ The helper that computes the SHA‑256 hash is named `_compute_sha256()`. The documentation and the unit‑test expectations refer to a method called `_sha256()`. | Tests (and future maintainers) look for `_sha256()`. The mismatch can cause confusion and may break tooling that introspects the class. |
| | 2️⃣ The `run()` method therefore calls the wrong private name (`self._compute_sha256`). | Even if the method works, the public contract is broken – callers expecting `_sha256()` won’t see it. |
| | 3️⃣ `client.table("import_batches").insert(batch_row)` passes a plain dict, while the Supabase Python client expects an *array* of rows (`[batch_row]`). | If the client enforces this shape, the batch insert will raise an error, halting the import pipeline. |
| | 4️⃣ The import‑batch insert does not explicitly include the `source_type` column in the returned payload (`result.data`). The code assumes `result.data` contains the inserted row with an `id`. | If the client returns a different shape (e.g. `{ “id”: … }`), the subsequent `raw_batch_id` extraction may fail, causing a runtime error. |
| **`mozc4med_dict/utils/kana.py`** | 5️⃣ The validation loop accepts **full‑width digits** (`0xFF10‑0xFF19`). The spec for `normalize_reading()` states that only *half‑width* digits (0‑9) are allowed; full‑width digits are retained only for import, not for export. | Allowing full‑width digits can produce illegal readings in the exported Mozc dictionary, violating the output contract. |
| **`scripts/manage_dict_enabled.py`** | 6️⃣ Only a `--disable` command‑line option is implemented. The top‑level CLAUDE.md mentions both enabling and disabling `dict_enabled` flags. | Users cannot re‑enable a term without editing the script, breaking the intended “toggle” behaviour. |
| | 7️⃣ The `list_abolished()` function prints raw rows without sorting or pagination. The spec does not demand this, but for large tables it can become unwieldy. | Minor usability issue; not a hard bug but worth improving. |
| **`mozc4med_dict/exporters/mozc_system_dict.py`** | 8️⃣ The RPC function name is stored in `_RPC_FUNCTION = "export_mozc_dict_entries"`. The top‑level CLAUDE.md (and Supabase migrations) refer to an RPC named `export_mozc_dict`. If the database uses a different function name, the exporter will fail. | Mismatch between code and DB schema leads to runtime errors when exporting. |
| | 9️⃣ The `_rows_from_rpc()` method logs a generic “unexpected payload type” error but does not include the offending data for debugging. | Reduces debuggability; not a failure but a quality gap. |
| **`mozc4med_dict/importers/ssk_shinryo_koi.py` & `ssk_shobyomei.py`** | 10️⃣ Field index constants (`_F_CHANGED_AT`, `_F_ABOLISHED_AT`, etc.) are duplicated across importers. If the SSK CSV layout changes, maintenance is error‑prone. | Not a bug now, but violates the DRY principle and can cause future bugs. |
| **`mozc4med_dict/utils/download.py`** | 11️⃣ The `urlretrieve` call has no timeout, and it suppresses SSL verification warnings (`# noqa: S310`). In CI environments without internet, this could hang. | Security and reliability concern. |
| **`scripts/*.py`** (all) | 12️⃣ Logging is set to INFO globally, which can flood the console with supabase client debug info. The spec suggests using `logging.basicConfig(level=logging.INFO, format="%(levelname)s %(message)s")` – fine, but adding a `--verbose` flag would give better control. | Minor UX improvement. |
| **General** | 13️⃣ No type‑hints on many public functions (e.g. `resolve_csv`, `export`, `manage_dict_enabled.main`). The repository targets `mypy --strict`. | Reduces static‑type‑checking coverage; could cause mypy failures. |
| | 14️⃣ The repository contains a `supabase` sub‑directory with migrations, but there is no script that automatically runs migrations before import/export. The spec mentions applying migrations on test projects. | Missing automation step; developers may forget to migrate, leading to schema mismatches. |
| | 15️⃣ The `tests/` suite expects the `BaseImporter._sha256()` name (see `tests/unit/test_importer_base.py`). Because the current code uses `_compute_sha256()`, the test suite may still pass (it never calls the private method directly) but future tests that introspect the method will fail. | Aligns with point 1 – a future‑proofing issue. |

### Summary of the most critical fixes

1. **Rename / alias `_compute_sha256` → `_sha256`** (and update the call in `run()`).
2. **Reject full‑width digits in `normalize_reading`** (or explicitly document the exception if needed).
3. **Add an `--enable` flag** to `manage_dict_enabled.py` (mirroring the disable logic).
4. **Correct Supabase batch insertion** to use a list (`[batch_row]`) and guard against missing `id`.
5. **Confirm the RPC name** used in the exporter matches the actual DB function (`export_mozc_dict` vs `export_mozc_dict_entries`).

Addressing these items will bring the codebase back into compliance with the CLAUDE.md specifications and improve reliability for both developers and automated CI runs.
